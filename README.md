# MTA Crypto - Multithreaded Brute Force Decryption

## Authors
- Noam Fishbain

## Overview
This project is a multithreaded application that simulates a brute-force password cracking scenario. The goal is to demonstrate thread synchronization between a producer (encrypter) and multiple consumers (decrypters), using mutexes and condition variables in C.

The encrypter thread generates a random password, encrypts it with a randomly generated key, and broadcasts the ciphertext to multiple decrypter threads. Each decrypter attempts to crack the password by guessing random keys and checking for printable plaintext. If a valid decryption is found, the result is sent back to the encrypter for validation.

## Features

- Dynamically configurable number of decrypter threads
- Adjustable password length
- Optional timeout for each cracking attempt
- Real-time logging of progress and results
- Thread-safe communication via `pthread` primitives

## Global Variables Explained

The following global variables are shared across the `encrypter` and `decrypter` threads to coordinate their actions. They reside in `cipher_manager.h` and are used throughout the program to manage synchronization and shared state.

### Synchronization Primitives

* `pthread_mutex_t g_mutex`
  A mutex used to ensure **mutual exclusion** when accessing or modifying shared resources like the ciphertext or plaintext guess.

* `pthread_cond_t g_new_cipher_cond`
  A condition variable used for **signaling** between threads.

  * The encrypter uses it to notify decrypters when a new password is ready.
  * Decrypters use it to notify the encrypter when a guess has been submitted.

---

### Shared State

* `char *g_ciphertext`
  Holds the **encrypted password** generated by the encrypter. Decrypter threads use this to attempt brute-force decryption.

* `size_t g_ciphertext_len`
  Length of the ciphertext in bytes. Necessary for proper memory allocation and cryptographic operations.

* `CrackResult *g_plaintext_candidate`
  A structure holding the **best guess** submitted so far by a decrypter thread. It includes:

  * `guess` â€“ the decrypted plaintext string.
  * `id` â€“ the ID of the thread that submitted the guess.

* `int g_password_cracked_or_timeout`
  A flag indicating whether the current password has already been cracked or a timeout has occurred.

  * `0` â†’ Still waiting for a valid guess or timeout
  * `1` â†’ Either a correct guess has been accepted or timeout has expired

---

### Configuration (Set at Startup)

* `int g_num_decrypters`
  Number of decrypter threads to be created. Configured via the `-n` or `--num-of-decrypters` command-line flag.

* `int g_password_len`
  Length of the generated plaintext password in bytes. Must be a **multiple of 8** to match the encryption block size. Set with `-l` or `--password-length`.

* `int g_timeout_secs`
  Optional timeout (in seconds) for each password round. If no valid decryption is received within this time, the encrypter generates a new password. Configured using the `-t` or `--timeout` flag. If not set, there is no timeout.

## Notes

- Passwords are generated using `MTA_get_rand_char()` and are guaranteed to be printable.
- Keys are 1/8 the length of the password.
- The system logs all major events including successful decryption, timeouts, and incorrect guesses.
- All the MTA functions (e.g., `MTA_encrypt`, `MTA_decrypt`, `MTA_get_rand_char`) were provided by our mentor, gavrielk.
